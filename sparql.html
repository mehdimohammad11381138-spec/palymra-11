SPARQL Queries

Technical Overview

This section documents the technical exploration of Wikidata and DBpedia. The queries are designed not only to extract data but also to test the limits of structured heritage documentation. We utilize the full range of SPARQL keywords to ensure academic rigor and data accuracy.

Query 1: Extract All Monuments in Palmyra

Objective: Verify the presence of Palmyra's primary monuments and their English descriptions.

Keyword used: DISTINCT — This keyword is essential to ensure that each monument IRI is returned only once, even if it is linked to multiple descriptions or labels.

PREFIX wd: <http://www.wikidata.org/entity/>
PREFIX wdt: <http://www.wikidata.org/prop/direct/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

SELECT DISTINCT ?item ?itemLabel ?description
WHERE {
  ?item wdt:P131 wd:Q5747 ;
        rdfs:label ?itemLabel ;
        schema:description ?description .
  FILTER (lang(?description) = "en")
}
ORDER BY ?itemLabel
LIMIT 20

Query 2: Temporal Data and Optional Properties

Objective: Extract construction and destruction dates to identify temporal gaps.

Keyword used: OPTIONAL — Since many monuments lack specific construction or destruction dates in Wikidata, OPTIONAL allows us to retrieve the available data without excluding monuments that are missing these fields.

PREFIX wd: <http://www.wikidata.org/entity/>
PREFIX wdt: <http://www.wikidata.org/prop/direct/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

SELECT DISTINCT ?monument ?monumentLabel ?constructionDate ?destructionDate
WHERE {
  ?monument wdt:P131 wd:Q5747 ;
           rdfs:label ?monumentLabel .
  OPTIONAL { ?monument wdt:P580 ?constructionDate . }
  OPTIONAL { ?monument wdt:P582 ?destructionDate . }
}
ORDER BY ?monumentLabel
LIMIT 50

Visual Proof: Temporal Gap Identification

Figure 5: Results in the Wikidata interface show that while destruction dates are often present, specific restoration timelines are missing.

Query 3: Pattern Matching with REGEX

Objective: Categorize monuments based on architectural type using string patterns.

Keyword used: FILTER(REGEX(...)) — We use regular expressions with the "i" flag (case-insensitive) to capture all variations of architectural terms like "Temple" or "Arch".

SELECT DISTINCT ?monument ?monumentLabel ?type
WHERE {
  ?monument wdt:P131 wd:Q5747 ;
           rdfs:label ?monumentLabel ;
           wdt:P31 ?type .
  FILTER (REGEX(?monumentLabel, "Temple|Arch|Theatre", "i"))
}
ORDER BY ?monumentLabel

Query 4: UNION — Combining Disparate Data States

Objective: Create a unified view of monuments that are either officially destroyed or designated as World Heritage.

Keyword used: UNION — This allows us to merge two different graph patterns: one for monuments with a destruction date property and another for those with a heritage designation property.

SELECT DISTINCT ?monument ?monumentLabel ?status
WHERE {
  {
    ?monument wdt:P131 wd:Q5747 ;
             rdfs:label ?monumentLabel ;
             wdt:P582 ?destructionDate .
    BIND("Destroyed" AS ?status)
    FILTER (YEAR(?destructionDate) >= 2015)
  }
  UNION
  {
    ?monument wdt:P131 wd:Q5747 ;
             rdfs:label ?monumentLabel ;
             wdt:P1435 wd:Q9259 .
    BIND("World Heritage" AS ?status)
  }
}
ORDER BY ?status ?monumentLabel

Query 5: Comprehensive Keyword Demonstration

Objective: A final complex query integrating DISTINCT, OPTIONAL, FILTER, REGEX, ORDER BY, and LIMIT.

SELECT DISTINCT ?monument ?monumentLabel ?constructionDate ?destructionDate
WHERE {
  ?monument wdt:P131 wd:Q5747 ;
           rdfs:label ?monumentLabel .
  OPTIONAL { ?monument wdt:P580 ?constructionDate . }
  OPTIONAL { ?monument wdt:P582 ?destructionDate . }
  FILTER (
    REGEX(?monumentLabel, "Temple|Arch|Theatre|Museum", "i") &&
    (!BOUND(?destructionDate) || YEAR(?destructionDate) >= 2015)
  )
}
ORDER BY ?monumentLabel
LIMIT 30
